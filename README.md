# AccessFlow

Smart. Secure. Seamless Visitor Management.

Production-ready monorepo for AccessFlow, a premium enterprise visitor management platform using a vanilla HTML/CSS/JavaScript frontend, Spring Boot backend, MongoDB Atlas, Cloudinary, and Render deployment.

## Tech Stack

- Frontend: HTML, CSS, vanilla JavaScript modules
- Backend: Spring Boot 3.5, Java 21, Maven
- Database: MongoDB Atlas via `MONGODB_URI`
- Storage: Cloudinary via `CLOUDINARY_URL` or individual Cloudinary credentials
- Deployment: Render static site and Docker-backed Spring Boot web service

## Local Development

### Backend

```bash
cd backend
./mvnw spring-boot:run
```

On Windows:

```powershell
cd backend
.\mvnw.cmd spring-boot:run
```

The API runs at `http://localhost:8080`.

If you have not configured MongoDB Atlas or a local MongoDB instance yet, start the API shell with:

```bash
./mvnw spring-boot:run -Dspring-boot.run.profiles=local
```

On Windows:

```powershell
.\mvnw.cmd spring-boot:run "-Dspring-boot.run.profiles=local"
```

### Frontend

The frontend is static. Open `frontend/index.html`, or serve it with any static server:

```bash
cd frontend
python -m http.server 3000
```

Then open `http://localhost:3000`.

For deployed static hosting, `frontend/assets/js/env.js` is generated by Render from `API_BASE_URL`.

## Backend Architecture

The backend uses layered packages:

- `controller/` REST boundaries
- `service/` business logic boundaries
- `repository/` MongoDB access
- `dto/` request/response contracts
- `entity/` Mongo documents and enums
- `config/` application, MongoDB, Cloudinary configuration
- `security/` JWT filter, token service, security chain
- `exception/` centralized API error handling

JWT role mapping and strict portal ownership are ready for:

- `SUPER_ADMIN`
- `ADMIN`
- `EMPLOYEE`
- `SECURITY_GUARD`

Each protected API namespace is guarded separately:

- `/api/v1/admin/**` requires `SUPER_ADMIN` or `ADMIN`
- `/api/v1/employee/**` requires `EMPLOYEE`
- `/api/v1/security/**` requires `SECURITY_GUARD`

Super admins receive full admin portal access and can create admin accounts. Admins receive visitor oversight, reports, and employee/security management through admin-owned endpoints instead of crossing into employee or security APIs. Authentication, authorization, DTO patterns, validation, pagination/search DTOs, and deployment wiring are implemented as the foundation.

## Enterprise Operations

- API versioning: all application APIs live under `/api/v1`; supported versions are exposed at `/api/versions`.
- Health: `/api/v1/health`, `/api/v1/health/live`, and `/api/v1/health/ready` support Render health checks and operational probes.
- Monitoring: Spring Boot actuator exposes health, info, metrics, and loggers. Production health details are hidden by default.
- OpenAPI: API docs are available at `/v3/api-docs`; Swagger UI is available at `/swagger-ui.html` outside production or when explicitly enabled.
- Security: JWT stateless auth, role-scoped namespaces, CORS allow-listing, request IDs, security headers, content security policy, HSTS, input sanitization, and global API rate limiting are enabled.
- CSRF: APIs are stateless bearer-token endpoints, so server-side CSRF sessions are disabled intentionally while CORS and authorization headers are enforced.
- Resilience: notification email delivery is asynchronous and retried; visitor expiry and notification retry schedulers are safe to run repeatedly.
- Caching: admin analytics and status summaries are cached and evicted on visitor lifecycle mutations.
- MongoDB: core visitor, notification, and audit query paths have explicit indexes; Atlas scheduled snapshots are the expected production backup mechanism.
- Logging: production logs are single-line structured console logs with request IDs for Render log drains.
- Deployment: `render.yaml` builds the backend with the Java 21 Dockerfile, starts it with the `prod` profile, JVM memory tuning, response compression, and readiness health checks.

## Frontend Portal Architecture

`frontend/index.html` is the authentication entry point. After login, the app validates that the JWT `roles` claim matches the returned session roles and redirects to the matching mini-application:

- `SUPER_ADMIN` -> `frontend/pages/admin/index.html`
- `ADMIN` -> `frontend/pages/admin/index.html`
- `EMPLOYEE` -> `frontend/pages/employee/index.html`
- `SECURITY_GUARD` -> `frontend/pages/security/index.html`

Each portal owns its sidebar, dashboard layout, route list, role guard invocation, role API calls, and role CSS. Shared utilities live under `frontend/js/shared/` and shared styling under `frontend/css/shared/`.

## Super Admin Bootstrap

AccessFlow does not ship with preset accounts or default passwords. On startup, if no `SUPER_ADMIN` or `ADMIN` exists, the backend creates one `SUPER_ADMIN` from Render environment variables only: `SUPER_ADMIN_NAME`, `SUPER_ADMIN_USERNAME`, `SUPER_ADMIN_EMAIL`, and `SUPER_ADMIN_PASSWORD`.

The bootstrap password is hashed with BCrypt before storage, is never logged, and is never exposed to the frontend. If an admin already exists, startup does not recreate the bootstrap account. `SUPER_ADMIN` accounts cannot be created through the public registration API.

## Render Deployment

`render.yaml` defines:

- `accessflow-api`: Docker-backed Spring Boot web service from `backend/`
- `accessflow-web`: static site from `frontend/`

The Blueprint uses Render's current static-site syntax: static frontends are declared as `type: web` with `runtime: static`. Do not change the frontend back to the old standalone static service type; Render rejects that older shape with `unknown type "static"`.

### 1. Create MongoDB Atlas

Create an Atlas cluster, add a database user, allow Render outbound access in Network Access, and copy the `mongodb+srv://...` connection string. Use a database name such as `visitor_management`.

### 2. Create Cloudinary

Create a Cloudinary project and copy either `CLOUDINARY_URL` or all three individual values: `CLOUDINARY_CLOUD_NAME`, `CLOUDINARY_API_KEY`, and `CLOUDINARY_API_SECRET`.

### 3. Create Render Services

Use Render's **New > Blueprint** flow, connect this repository, and let Render read `render.yaml`. The backend builds from `backend/Dockerfile` with Java 21, auto-deploys from the connected branch, and runs with the `prod` Spring profile through `SPRING_PROFILES_ACTIVE=prod`. The static frontend publishes the `frontend/` folder directly; do not add a catch-all rewrite because the portals are real static pages under `/pages/...`.

### 4. Set Backend Environment Variables

- `MONGODB_URI`
- `JWT_SECRET` with at least 32 random characters
- `BACKEND_PUBLIC_URL` to the deployed backend origin
- `SUPER_ADMIN_NAME`, `SUPER_ADMIN_USERNAME`, `SUPER_ADMIN_EMAIL`, and `SUPER_ADMIN_PASSWORD` for first-start bootstrap
- `CLOUDINARY_URL`, or `CLOUDINARY_CLOUD_NAME` + `CLOUDINARY_API_KEY` + `CLOUDINARY_API_SECRET`
- `CORS_ALLOWED_ORIGINS` to the deployed frontend URL
- `SENDGRID_API_KEY` and `SENDGRID_FROM_EMAIL` when `SENDGRID_ENABLED=true`

The backend now fails startup in the `prod` profile when required production values are missing, local defaults are still active, Cloudinary is not configured, or production CORS still points at localhost.

### 5. Set Frontend Environment Variables

`render.yaml` provides a default `API_BASE_URL` for the `accessflow-api` service name. If you rename the backend service or Render assigns a different URL, update `API_BASE_URL` on the static site to the deployed backend URL including `/api/v1`, for example:

```text
https://accessflow-api.onrender.com/api/v1
```

Set `FRONTEND_PUBLIC_URL` to the deployed static site origin. The frontend build command writes `API_BASE_URL` to `frontend/assets/js/env.js`; if it is missing, the static deploy fails instead of silently pointing users to localhost.

### 6. Verify Production

After deployment:

- Open `https://<api-service>.onrender.com/api/v1/health/ready` and confirm it returns `UP`.
- Open the frontend root URL and sign in.
- Upload a visitor photo to verify Cloudinary.
- Register a visitor and confirm MongoDB Atlas receives the record.
- Confirm browser devtools show no console errors or failed API calls.
